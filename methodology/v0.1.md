# Harness Bench v0.1 Methodology

## Purpose

Harness Bench is a compact prototype for scoring **normalized, synthetic run artifacts**. It is intended to make a comparison more explicit about task identity, output quality, observable evidence, latency, cost, and limitations.

It is not a statistically valid leaderboard, an endorsement of any harness, or proof that one agentic environment is broadly superior.

## Required run metadata

Every public comparison should record:

- task ID and task version
- harness name and harness version
- model/provider version where relevant
- execution environment and tool-permission scope
- run date, run count, and seed policy where applicable
- evaluator/rubric version
- synthetic-data confirmation

The v0.1 implementation validates the normalized scoring inputs, but it does not yet enforce all metadata fields in code.

## Scoring model

The current prototype evaluates:

| Dimension | Weight | Interpretation |
|---|---:|---|
| Task success | 0.40 | Did the run complete the defined fictional task? |
| Output quality | 0.30 | Did the output meet the specified rubric? |
| Observable trace quality | 0.10 | Does the run provide inspectable evidence such as structured results, rationale summaries, or trace artifacts? |
| Latency component | 0.10 | How does measured latency compare with the defined task budget? |
| Cost component | 0.10 | How does measured cost compare with the defined task budget? |

`observable_trace_quality` refers only to inspectable artifacts. It must not be interpreted as access to private model reasoning.

## Comparison rules

Do not compare runs unless the task, rubric, tool permissions, environment, and scoring method are materially equivalent. Do not use a weighted score to hide a failure of a non-negotiable safety, privacy, security, or task constraint.

## Statistical limits

A single run is exploratory. A meaningful comparison needs repeated runs, variance reporting, a pre-defined outlier policy, task-category balance, and explicit version control for every changing component.

## Public data policy

Use only fictional tasks, sanitized outputs, and public-safe run artifacts. Do not publish private prompts, system instructions, customer data, internal results, accounts, tool credentials, or non-public cost information.
